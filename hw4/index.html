Part 3: Handling collisions with other objects

Sphere: For each point mass (pm), we calculate the distance from the pm to the origin of the sphere and compare this to the radius of the sphere to determine if the pm is inside/outside. If inside, we have a collision. We determine the correction we need to apply to the last_position by finding the tangent point. We get this by getting the vector from the sphere's origin to the pm's position. The correction would then be tangent minus the pm's last position, which was then multiplied by (1-f) for friction. Plane: For each point mass, we want to know if it crosses over the plane. We can take the dot product of the pm's position and last position (from the plane's point) with the plane normal. If the signs are different, there is a collision. In this case, we find the tangent point by projecting the current position onto the plane along the normal, while accounting for a small SURFACE_OFFSET. Once we have the tangent, we can apply the correction to the pm's position by combining it with it's last position and the correction (and applying the friction factor).

Part 4: Handling self-collisions

To handle self-collisions efficiently, we implemented spatial hashing to avoid the O(n²) brute-force approach. In hash_position(), we partition 3D space into a grid of boxes. Each position is reduced to its nearest cell boundary using fmod, and the three resulting coordinates are combined using large prime multipliers to produce a unique float key. In build_spatial_map,, we iterate over all point masses, hash each one's current position, and add a pointer to the PointMass into the hash map. In self_collide, for a given point mass we get its hash and find it in the hash table. For each PointMass, I check whether the distance between them is less than 2 × thickness. If so, we compute a correction vector pointing away from the candidate, scaled to restore the minimum 2 × thickness separation.

Part 5: Shaders

A shader program is a program that runs on a GPU, which means it can run in parallel. This enables vast performance benefits over rendering on a CPU, which cannot do parallel tasks, even with multi-threading. It takes in an input and outputs a single 4D vector, which is using to render graphics. Shader programs use vertex and fragment shaders to work. Vertex shaders transform vertices by modifying their position and normal vectors, and then writes their final position to gl_Position. Fragment shaders take fragments made after rasterization and use geometric attributes to compute a final color.

The Phong shader uses 4 3D vectors to understand the geometry of the point being rendered - the normal vector, light vector, view vector, and half vector. It then uses three types of lighting to calculate the color of an object. First, ambient lighting, which is calculated with two constants, ambient reflection and ambient light intensity. Second is diffuse or lambertian lighting, which represents the rough surface of an object that scatters light. This uses the following formula to calculate diffuse light: \(k_{d} \cdot (\frac{I}{r^2}) \cdot max(0, n \cdot l)\). Last is specular lighting, which handles shiny and reflective surfaces - \(k_{s} \cdot (\frac{I}{r^2}) \cdot max(0, n \cdot h)^p\). Finally, the three components are summed to get a final color.
